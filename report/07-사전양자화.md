# 07 — 사전 양자화 4-bit (int4/AWQ/NVFP4)

> **무엇을·왜:** 1부 정밀도 축은 bf16/fp8/int8/qat까지였고 모두 무손실로 확인됐다. 미답 영역 둘을 채운다 — ① **fp8 아래(4-bit)에서 무손실이 깨지는가**, ② **dense 모델에서 4-bit가 Marlin 대역폭 절감으로 fp8 속도를 이기는가**. online(동적) 양자화가 아니라 **HF에 publish된 pre-quantized 슬림 체크포인트**로만 측정(실제 배포되는 형태). 배터리는 1부와 동일(speed + GPQA full 198 + MMLU-Pro 500 + KMMLU 500, thinking-on). 데이터: `../results_consolidated.csv`.
>
> **HF 가용성(2026-06):** 공식 Qwen3.6 GPTQ/AWQ는 **부재(404)**. 그래서 4-bit는 **커뮤니티 리포 + 공식 NVIDIA NVFP4**로 구성 — 출처를 표에 명시한다(권위 부풀리기 금지).

## B. 해석

1. **정확도 — 4-bit도 무손실 (fp8 대비 노이즈 이내).** qwen35-int4(GPTQ)·qwen35-awq 모두 GPQA/MMLU-Pro/KMMLU가 fp8과 ±0.01 내, gemma31B-nvfp4는 MMLU-Pro가 fp8과 동일(0.862)·GPQA −0.006, qwen27-awq는 MMLU-Pro가 오히려 +0.032. 4-bit 4종 전부 fp8급 → **4-bit에서도 무손실 경계는 안 깨진다.**
   - **양자화기 무관(교차검증):** 같은 Qwen35-A3B를 **GPTQ와 AWQ 두 양자화기로 독립 측정 → GPQA 0.793 완전 일치**(MMLU 0.858/0.852, KMMLU 0.578/0.592). "무손실"이 한 리포·한 방식의 우연이 아님을 확인.
   - **KMMLU만 소폭↓**(NVFP4 −0.038, qwen27 −0.030) — 한국어 지식에서 4-bit 손실이 가장 먼저 비치나 여전히 작음.

2. **속도 — "dense 4-bit가 fp8을 이긴다"는 정수 W4A16만 참.** Gemma31B에서 **qat-w4a16(int4)=90 tok/s로 fp8(68)을 압도**하지만, **NVFP4=65로 fp8과 비등(8K는 50<62로 더 느림)**. MoE(qwen35-int4)도 208.9로 fp8 217보다 −4%. 이유:
   - **NVFP4는 H100에서 가속 안 됨** — vLLM 로그가 명시: *"GPU does not have native support for FP4 computation ... weight-only FP4 ... leveraging the Marlin kernel. This may degrade performance for compute-heavy workloads."* native FP4 텐서코어는 **Blackwell 전용**이라 H100은 FP4→고정밀 **디퀀트 에뮬**(MarlinNvFp4) → 대역폭 이득을 오버헤드가 상쇄.
   - **MoE int4**는 GPTQ가 attention·gate를 16-bit로 남겨(활성 3B) 4-bit footprint·트래픽 이득이 작아 fp8 텐서코어를 못 이김.
   - → **H100에서 빠른 4-bit = 정수 W4A16(qat/GPTQ-Marlin) on dense.** NVFP4의 강점(Blackwell 네이티브 FP4 속도)은 H100에서 드러나지 않는다.

3. **메모리 — 4-bit의 실이득은 속도가 아니라 VRAM.** qwen35 weight 23G(int4) vs 35G(fp8) = ~34%↓(MoE라 attn·gate 16-bit 유지로 절반은 아님). 같은 카드에 더 긴 컨텍스트·더 큰 모델을 올릴 때 값어치.

4. **MTP는 4-bit 위에서도 가속 + 무손실.** embedded MTP head를 켜자 **qwen35-int4 +49%/+59%(209→311)**, **qwen27-awq +89%/+80%(86→163)** — spec-decode가 양자화와 직교적으로 쌓인다(dense-hybrid가 더 큰 이득). GPQA는 AR 대비 무손실(int4 0.783/0.793, awq 0.813/0.803). 즉 **MoE int4가 AR로는 fp8 속도를 못 이겨도, MTP를 얹으면 fp8 AR(217)을 크게 추월**(311). 단 MTP 서빙은 KV 여유가 필요(아래 C 교훈).

**한 줄 결론:** 사전 양자화 4-bit(GPTQ/AWQ/NVFP4)는 **fp8 무손실 대체이되 AR 이점은 VRAM 절감**(속도 이득은 H100선 정수 W4A16 dense에서만, NVFP4는 Blackwell 대기). **여기에 MTP를 얹으면 4-bit가 속도까지 fp8을 추월**한다.

### 방법론 교훈
- **재패키징 체크포인트는 chat_template를 빠뜨릴 수 있다.** palmfuture GPTQ는 tokenizer_config의 chat_template가 비어(len 0) thinking이 안 켜졌다 — 가중치만 양자화됐고 아키는 동일하므로 **공식 베이스의 템플릿을 주입**하면 동등성 회복. detokenize로 프롬프트가 `...assistant\n<think>\n`으로 끝나는지 확인이 결정적.
- **양자화 메타는 커널을 결정한다.** vLLM이 quant_method를 읽어 자동으로 Marlin/awq_marlin/MarlinNvFp4를 고름 — GPTQ/AWQ/NVFP4 모두 cudagraph 적재 성공(폴백 없음). NVFP4만 kv-cache fp8이 메타에 박혀 자동 적용.
- **검증은 합성 curl이 아니라 실제 하네스로.** max_tokens 작은 단건 curl은 `</think>` 미도달로 파서가 출력을 삼켜 오해를 부른다 — lm-eval GPQA limit=5의 flexible-extract 비영(非零)으로 추출 정상을 실증.
- **출처 구분.** NVFP4만 공식(nvidia), 나머지 int4/AWQ는 커뮤니티 — "무손실"은 해당 커뮤니티 체크포인트 기준임을 명시. GPTQ vs AWQ(qwen35) 일치로 양자화기 무관성을 교차검증.
- **MTP 서빙 = `--max-model-len 40960` 필수.** MTP를 16384로 띄우면 lm-eval `max_gen_toks=32768` 요청이 `400 (max_tokens>max_model_len)`을 받고, lm-eval async 세션이 그 오류로 무너져 **`RuntimeError: Session is closed`가 전 요청에 연쇄**(0점). len을 출력버짓 이상으로. 또 MTP 부하에선 thinking 프록시(:8001, 동기 서버)가 연결을 끊어 같은 증상 → **GPQA는 :8000 직결**(템플릿이 enable_thinking 미지정 시 기본 `<think>` 부착이라 thinking-on 유지).

## D. 1부 대비 종합
- **정밀도 축 완성:** bf16 → fp8 → int8 → qat(w4a16) → **4-bit(GPTQ/AWQ/NVFP4)** 전 구간 정확도 무손실 확인.
- **운영 함의:** 무손실이 목표면 **fp8이 기본**(속도·안정), **VRAM이 빠듯하면 정수 4-bit(qat/GPTQ)**가 정확도 유지하며 메모리↓·(dense면)속도↑. **NVFP4는 H100에선 메리트 없음**(Blackwell 대기).
